# スキル (.claude/skills/, Agent Skills) 編集ガイドライン

## スキルの役割

スキルはオンデマンドで読み込まれる拡張指示。
ユーザーが `/skill-name` で呼び出すか、Claude が関連性を判断して自動で読み込む。

常に適用すべき規約・方針にはルール (.claude/rules/) を使う。

## ディレクトリ構造

```
.claude/skills/<skill-name>/
├── SKILL.md           # メイン指示 (必須、500行以下を目安)
├── references/        # 詳細なリファレンス (必要時に読み込み)
├── scripts/           # 実行可能スクリプト
└── examples/          # 出力例
```

サポートファイルは SKILL.md から参照し、読み込みタイミングを明示する。

### Progressive Disclosure (段階的読み込み)

スキルは3層で読み込まれる:

1. メタデータ: (name + description) — 常にコンテキストに存在 (~100語)
2. SKILL.md 本体: — スキル呼び出し時に読み込み (500行以下を目安)
3. バンドルリソース: — 必要時にオンデマンドで読み込み (サイズ制限なし)

SKILL.md が肥大化したら、詳細をリファレンスファイルに分離し、
SKILL.md からは「いつ・何を読むか」のポインタだけ残す。
300行超のリファレンスには目次を付ける。

ドメインが複数ある場合はリファレンスを分離する:

```
cloud-deploy/
├── SKILL.md          # ワークフローと選択ロジック
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

## 配置場所

| 場所         | パス                               | 適用範囲             |
| ------------ | ---------------------------------- | -------------------- |
| ユーザー     | `~/.claude/skills/<name>/SKILL.md` | 全プロジェクト       |
| プロジェクト | `.claude/skills/<name>/SKILL.md`   | そのプロジェクトのみ |

## YAML frontmatter

```yaml
---
name: my-skill # /my-skill で呼び出し (省略時はディレクトリ名)
description: 何をするか、いつ使うか # トリガー判定に使われる (重要)
---
```

主要フィールド:

- `name`: スキル名。小文字・数字・ハイフンのみ、最大64文字
- `description`: スキルの説明。Claude が自動トリガーするかの判定に直結する。
  Claude はスキルを「使わない」方向に傾きがちなので、やや積極的に書く。
  ユーザーが自然に使うキーワードや、トリガーすべき具体的な状況を含める
  - NG: 「ダッシュボードを作成する」
  - OK: 「社内データを表示するシンプルなダッシュボードを作成する。
    ユーザーがダッシュボード、データ可視化、社内メトリクスに言及したら使用する」
- `disable-model-invocation: true`: `/name` でのみ呼び出し可能にする。
  デプロイなど副作用のあるワークフロー向け
- `user-invocable: false`: Claude のみが呼び出し可能。バックグラウンド知識向け
- `context: fork`: サブエージェントで実行。会話履歴にアクセスしない
- `agent`: `context: fork` 時のサブエージェントタイプ (例: `Explore`, `Plan`)
- `allowed-tools`: スキル実行時に許可なしで使えるツールを制限

## 変数

- `$ARGUMENTS`: 呼び出し時の全引数
- `$ARGUMENTS[N]` / `$N`: N番目の引数 (0始まり)
- `${CLAUDE_SKILL_DIR}`: SKILL.md のあるディレクトリパス
- `${CLAUDE_SESSION_ID}`: セッションID
- `` !`command` ``: 読み込み前にシェルコマンドを実行し、出力で置換

## 記述のコツ

- 指示は命令形で書く
- 「なぜそうするか」を説明し、MUSTやNEVERの多用を避ける。
  モデルは理由を理解すると想定外のケースにも適切に対応できる
- 出力フォーマットはテンプレートで明示する:

  ```markdown
  ## レポート構造

  # [タイトル]

  ## 概要

  ## 主な発見

  ## 推奨事項
  ```

- 具体例を含める (入力→出力の対で示すと効果的)
- テスト実行で繰り返し生成されるスクリプトは scripts/ にバンドルする
- 特定の例にだけ効く狭い指示は避け、汎用的に書く
